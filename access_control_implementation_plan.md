# План внедрения системы ролевого доступа для Facebook Ads Telegram Bot

## 1. Общее описание системы доступа

Система ролевого доступа будет включать три основные роли пользователей с различными уровнями привилегий:

1. **Администратор**: полный доступ ко всем функциям и данным бота.
2. **Таргетолог**: полный доступ к функциям управления кампаниями без возможности управления платежными данными.
3. **Партнер**: ограниченный доступ только к статистике определенных кампаний, назначенных администратором.

## 2. Рекомендации по реализации

### 2.1. Принципы инкрементальной разработки

**ВАЖНО:** Внедрение системы ролевого доступа должно происходить небольшими инкрементальными шагами с обязательным тестированием после каждого шага.

1. **Маленькие коммиты**
   - Каждый коммит должен содержать одно атомарное изменение
   - Размер коммита не должен превышать 200-300 строк кода
   - Избегать объединения нескольких логических изменений в один коммит

2. **Частое тестирование**
   - Проводить тестирование после каждого существенного изменения
   - Писать юнит-тесты для всех новых функций и классов
   - Проверять работоспособность интерфейса после каждого изменения, влияющего на пользовательские взаимодействия

3. **Поэтапное внедрение**
   - Сначала реализовать базовую модель данных и миграции
   - Далее добавить логику авторизации без интерфейса
   - Затем разработать административный интерфейс
   - В последнюю очередь реализовать пользовательский интерфейс для партнеров

4. **Обратная совместимость**
   - Все изменения должны сохранять обратную совместимость
   - Существующие пользователи не должны заметить изменений, пока не будет включена система ролей
   - Добавить функцию-переключатель для отключения новой функциональности в случае проблем

### 2.2. Порядок реализации подзадач

Для каждого этапа рекомендуется следующий порядок реализации:

1. **Подготовка базы данных**
   - Реализовать и проверить миграции в тестовой среде
   - Протестировать работу с данными с помощью простых скриптов
   - Только после успешного тестирования применять в основной базе

2. **Разработка бизнес-логики**
   - Начать с реализации базовых моделей и репозиториев
   - Добавить функции проверки прав и ограничения
   - Протестировать логику в изоляции от интерфейса

3. **Разработка интерфейса**
   - Начать с простых компонентов и постепенно добавлять сложность
   - Проводить ручное тестирование каждого компонента
   - Фокусироваться сначала на корректности, затем на юзабилити

4. **Интеграционное тестирование**
   - После реализации каждой подсистемы проводить интеграционное тестирование
   - Тестировать взаимодействие компонентов в разных сценариях
   - Проверять граничные условия и обработку ошибок

## 4. Изменения в структуре базы данных

### 4.1. Модели данных

#### Таблица User (расширение существующей)
```sql
ALTER TABLE User ADD COLUMN role VARCHAR(20) NOT NULL DEFAULT 'partner';
```

#### Новая таблица AccessControl
```sql
CREATE TABLE AccessControl (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    partner_id INTEGER NOT NULL,
    admin_id INTEGER NOT NULL,
    campaign_id VARCHAR(255) NOT NULL,
    granted_at DATETIME NOT NULL,
    expires_at DATETIME NULL,
    is_active BOOLEAN NOT NULL DEFAULT 1,
    FOREIGN KEY (partner_id) REFERENCES User(id),
    FOREIGN KEY (admin_id) REFERENCES User(id)
);
```

#### Новая таблица AccessRequest
```sql
CREATE TABLE AccessRequest (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    partner_id INTEGER NOT NULL,
    requested_at DATETIME NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending', -- 'pending', 'approved', 'rejected'
    processed_by INTEGER NULL,
    processed_at DATETIME NULL,
    FOREIGN KEY (partner_id) REFERENCES User(id),
    FOREIGN KEY (processed_by) REFERENCES User(id)
);
```

## 6. Логика работы с партнерами

### 6.1. Процесс запроса доступа партнером
1. Партнер запрашивает статистику через команду или кнопку
2. Система создает запрос доступа в базе данных
3. Администраторы получают уведомление о новом запросе
4. После одобрения партнер получает доступ к статистике указанных кампаний

### 6.2. Панель управления доступом для администраторов
1. Просмотр всех запросов доступа
2. Одобрение/отклонение запросов
3. Назначение конкретных кампаний для просмотра партнером
4. Настройка срока действия доступа
5. Отзыв доступа

## 9. Риски и их митигация

### 9.1. Технические риски
- **Проблема:** Разрешения не применяются корректно
  **Решение:** Внедрение тщательного логирования и мониторинга доступа

- **Проблема:** Производительность при множестве проверок доступа
  **Решение:** Оптимизация запросов и кеширование прав доступа

### 9.2. Пользовательские риски
- **Проблема:** Партнеры не понимают ограничения доступа
  **Решение:** Разработать понятные сообщения об ошибках и руководство

- **Проблема:** Администраторы могут неправильно настроить разрешения
  **Решение:** Создание простого и понятного интерфейса управления доступом с подтверждениями
